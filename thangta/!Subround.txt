REAL-TIME SCORING SYSTEM (SYNCHRONIZED SUBROUND VERSION – REFINED)

The system follows a strict lifecycle:

Capture → Wait → Compute → Display → Finalize → Flush

All scoring is handled in-memory using nested dictionaries, and each subround is processed only when fully complete.

PHASE 1: CAPTURE (INPUT COLLECTION)

Each scorer submits:

match_id
round
subround
corner (blue/red)
scorer_id
score

The system stores the input in memory:

scores[match_id][round][subround][corner][scorer_id] = score

Rules:

If a scorer resubmits, the previous value is overwritten
Each scorer can only have one score per subround per corner
Data may arrive out of order and must still be accepted
PHASE 2: WAIT (COMPLETION CHECK)

After every submission, the system checks:

“Do we have all 3 scorers for this subround and corner?”

Condition:

 

 

Mark subround as PENDING if nothing 
Do NOT compute anything
Do NOT expose partial results (important for consistency)
PHASE 3: COMPUTE (ON COMPLETION ONLY)

Once all 3 scores are available:

Retrieve the three scores
Compute the average:

avg = (s1 + s2 + s3) / 3

Store the computed result:

liveState[match][round][subround][corner] contains:

list of scores
computed average
status set to COMPLETE
CRITICAL SAFETY RULE (YOU MUST KEEP THIS)

Before computing:

Check if this subround is already marked COMPLETE
If yes → skip computation

This prevents duplicate calculations when submissions arrive close together.

Update running total:

total[match][round][corner] is increased by the computed average

PHASE 4: DISPLAY (POLLING-BASED UI)

The UI polls the system every 1–2 seconds:

GET /match/{id}/live

The response includes:

Completed subrounds with averages
Pending subrounds WITHOUT exposing partial scores
DISPLAY RULES
If a subround is incomplete:
→ Show “WAITING FOR SCORES”


ORDER ENFORCEMENT  

Subrounds must be displayed in order.

If subround N is not complete:
→ Subround N+1 must NOT be displayed, even if complete

This ensures consistency and prevents confusion.

PHASE 5: FINALIZE

At the end of the match:

Sum all completed subround averages:

final_score = sum of all subround averages

Store ONLY the final result in database (optional)
PHASE 6: FLUSH

After match completion:

Delete all in-memory data:

delete scores[match_id]
delete liveState[match_id]
delete total[match_id]

This prevents memory buildup.

KEY RULES
No averaging until all 3 scorers submit
No null values allowed in final computation
Subround is atomic (all-or-nothing)
System must handle out-of-order submissions
Subrounds must be shown in strict sequence
Computation must happen only once per subround
